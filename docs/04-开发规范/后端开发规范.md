## 9. 日志规范

### 9.1 日志库选择
项目使用 **loguru** 作为主要日志库，它比Django原生logging更简洁、功能更强大。

#### 9.1.1 loguru优势
- **简洁的API**: 无需复杂配置，开箱即用
- **自动日志轮转**: 内置文件大小和时间轮转
- **结构化日志**: 原生支持JSON格式
- **异步日志**: 高性能异步写入
- **异常追踪**: 更好的异常信息展示
- **彩色输出**: 开发环境友好的彩色日志

### 9.2 loguru配置
```python
# config/logging.py
from loguru import logger
import sys
import os

def setup_logging():
    """配置loguru日志系统"""
    
    # 移除默认处理器
    logger.remove()
    
    # 开发环境：控制台输出（彩色）
    if os.getenv('DEBUG', 'False').lower() == 'true':
        logger.add(
            sys.stdout,
            format="<green>{time:YYYY-MM-DD HH:mm:ss}</green> | "
                   "<level>{level: <8}</level> | "
                   "<cyan>{name}</cyan>:<cyan>{function}</cyan>:<cyan>{line}</cyan> | "
                   "<level>{message}</level>",
            level="DEBUG",
            colorize=True
        )
    
    # 生产环境：文件输出
    logger.add(
        "logs/app_{time:YYYY-MM-DD}.log",
        format="{time:YYYY-MM-DD HH:mm:ss} | {level: <8} | {name}:{function}:{line} | {message}",
        level="INFO",
        rotation="00:00",  # 每天轮转
        retention="30 days",  # 保留30天
        compression="zip",  # 压缩旧日志
        encoding="utf-8"
    )
    
    # 错误日志单独记录
    logger.add(
        "logs/error_{time:YYYY-MM-DD}.log",
        format="{time:YYYY-MM-DD HH:mm:ss} | {level: <8} | {name}:{function}:{line} | {message}",
        level="ERROR",
        rotation="00:00",
        retention="90 days",  # 错误日志保留更久
        compression="zip",
        encoding="utf-8"
    )
    
    # JSON格式日志（用于日志分析）
    logger.add(
        "logs/app_{time:YYYY-MM-DD}.json",
        format=lambda record: {
            "timestamp": record["time"].isoformat(),
            "level": record["level"].name,
            "module": record["name"],
            "function": record["function"],
            "line": record["line"],
            "message": record["message"],
            "extra": record["extra"]
        },
        level="INFO",
        rotation="00:00",
        retention="30 days",
        compression="zip",
        serialize=True  # JSON序列化
    )

# settings/base.py
from config.logging import setup_logging

# 初始化日志配置

```

### 9.5 性能优化配置
```python
# config/logging.py
from loguru import logger
import asyncio

def setup_async_logging():
    """配置异步日志以提高性能"""
    
    # 异步文件处理器
    logger.add(
        "logs/async_{time}.log",
        level="INFO",
        rotation="100 MB",
        retention="30 days",
        compression="zip",
        enqueue=True,  # 启用异步队列
        backtrace=True,  # 启用异常回溯
        diagnose=True   # 启用变量诊断
    )

# 批量日志处理
class BatchLogger:
    """批量日志处理器"""
    
    def __init__(self, batch_size: int = 100):
        self.batch_size = batch_size
        self.log_buffer = []
    
    def add_log(self, level: str, message: str, **kwargs):
        """添加日志到缓冲区"""
        self.log_buffer.append({
            'level': level,
            'message': message,
            'extra': kwargs
        })
        
        if len(self.log_buffer) >= self.batch_size:
            self.flush()
    
    def flush(self):
        """刷新缓冲区"""
        for log_entry in self.log_buffer:
            getattr(logger, log_entry['level'].lower())(
                log_entry['message'],
                **log_entry['extra']
            )
        self.log_buffer.clear()
```

### 9.6 Django集成配置
```python
# settings/base.py
import logging
from loguru import logger

class InterceptHandler(logging.Handler):
    """拦截Django日志并转发给loguru"""
    
    def emit(self, record):
        # 获取对应的loguru级别
        try:
            level = logger.level(record.levelname).name
        except ValueError:
            level = record.levelno

        # 查找调用者
        frame, depth = logging.currentframe(), 2
        while frame.f_code.co_filename == logging.__file__:
            frame = frame.f_back
            depth += 1

        logger.opt(depth=depth, exception=record.exc_info).log(
            level, record.getMessage()
        )

# 配置Django使用loguru
logging.basicConfig(handlers=[InterceptHandler()], level=0, force=True)

# 禁用Django默认日志配置
LOGGING_CONFIG = None
```

## 12. 版本控制

### 12.1 分支策略
- `main`: 主分支，生产环境代码
- `develop`: 开发分支
- `feature/*`: 功能分支
- `hotfix/*`: 热修复分支

### 12.2 提交规范

#### 12.2.1 提交格式
Git commit 要清晰明了的说明本次提交的目的。

**格式**: `[type](scope): subject`

**示例**:
- `[feature](family): add quality field`
- `[bugfix](auth): fix user login validation`

#### 12.2.2 提交类型（Type）
[强制] type 用于说明 git commit 的类别，只允许使用下面的标识：

- **[feature]**: 新功能（feature）
- **[bugfix]**: 修复bug，可以是QA发现的BUG，也可以是研发自己发现的BUG
  - [bugfix]: 产生diff并自动修复此问题。适合于一次提交直接修复问题
  - to: 只产生diff不自动修复此问题。适合于多次提交。最终修复问题提交时使用fix
- **[docs]**: 文档（documentation）
- **[format]**: 格式（不影响代码运行的变动）
- **[refactor]**: 重构（即不是新增功能，也不是修改bug的代码变动）
- **[performance]**: 优化相关，比如提升性能、体验
- **[test]**: 增加测试
- **[chore]**: 构建过程或辅助工具的变动
- **[migrate]**: 代码迁移

#### 12.2.3 作用域（Scope）
scope 用于说明 commit 影响的范围，比如数据层、控制层、视图层等等，视项目不同而不同。

#### 12.2.4 主题（Subject）
[强制] subject是commit目的的简短描述，不超过50个字符：
- 优先使用英文，也可以使用中文
- 结尾不需要标点符号

#### 12.2.5 提交方式
- 所有项目的 MR 都强制要求 MR 方式为 rebase & squash，所以每个 MR 提交到默认分支之后，只会有 1 个 commit
- 大家务必一个 MR 只做一件事，一件事只提一个 MR

#### 12.2.6 分支命名规范
- 把分支名定义为 `feature/{模块}/add-user` 的样式
- 把 MR 和 commit 的 title 定义为 `[feature](模块): 功能描述` 的格式

#### 12.2.7 提交示例
```
[feature](family): 添加族谱可视化功能
[bugfix](auth): 修复用户登录验证问题
[performance](api): 优化族谱查询性能
[refactor](models): 重构用户模型结构
[format]: 代码格式化调整
[docs]: 更新API文档
[test]: 添加用户认证单元测试
[chore]: 更新依赖包版本
```

#### 12.2.8 代码规范要求
- 单元测试覆盖率 ≥ 70%

#### 12.2.9 代码审查
- 所有 MR Approver，请务必严格按照上述要求作 Code Review
