#### 3.2.3 视图定义
```python
from ninja import Router
from django.shortcuts import get_object_or_404
from .models import User
from .serializers import UserSchema, UserCreateSchema

router = Router()

@router.get("/users", response=List[UserSchema])
def list_users(request):
    """获取用户列表"""
    return User.objects.all()

@router.post("/users", response=UserSchema)
def create_user(request, payload: UserCreateSchema):
    """创建用户"""
    user = User.objects.create(**payload.dict())
    return user

@router.get("/users/{user_id}", response=UserSchema)
def get_user(request, user_id: int):
    """获取用户详情"""
    return get_object_or_404(User, id=user_id)
```

### 5.3 索引设计
```python
class FamilyMember(BaseModel):
    # 使用BigIntegerField进行逻辑关联，不使用ForeignKey
    family_id = models.BigIntegerField(help_text='家族ID，逻辑关联families.id')
    user_id = models.BigIntegerField(help_text='用户ID，逻辑关联users.id')
    role = models.CharField(max_length=50)
    
    class Meta:
        db_table = 'family_members'
        indexes = [
            models.Index(fields=['family_id', 'user_id']),
            models.Index(fields=['family_id', 'role']),
            models.Index(fields=['created_at']),
        ]
        unique_together = ['family_id', 'user_id']
    
    @property
    def family(self):
        """获取关联的家族对象"""
        from apps.families.models import Family
        return Family.objects.get(id=self.family_id)
    
    @property
    def user(self):
        """获取关联的用户对象"""
        from django.contrib.auth import get_user_model
        User = get_user_model()
        return User.objects.get(id=self.user_id)
```

## 12. 版本控制

### 12.1 分支策略
- `main`: 主分支，生产环境代码
- `develop`: 开发分支
- `feature/*`: 功能分支
- `hotfix/*`: 热修复分支

### 12.2 提交规范

#### 12.2.1 提交格式
Git commit 要清晰明了的说明本次提交的目的。

**格式**: `[type](scope): subject`

**示例**:
- `[feature](family): add quality field`
- `[bugfix](auth): fix user login validation`

#### 12.2.2 提交类型（Type）
[强制] type 用于说明 git commit 的类别，只允许使用下面的标识：

- **[feature]**: 新功能（feature）
- **[bugfix]**: 修复bug，可以是QA发现的BUG，也可以是研发自己发现的BUG
  - [bugfix]: 产生diff并自动修复此问题。适合于一次提交直接修复问题
  - to: 只产生diff不自动修复此问题。适合于多次提交。最终修复问题提交时使用fix
- **[docs]**: 文档（documentation）
- **[format]**: 格式（不影响代码运行的变动）
- **[refactor]**: 重构（即不是新增功能，也不是修改bug的代码变动）
- **[performance]**: 优化相关，比如提升性能、体验
- **[test]**: 增加测试
- **[chore]**: 构建过程或辅助工具的变动
- **[migrate]**: 代码迁移

#### 12.2.3 作用域（Scope）
scope 用于说明 commit 影响的范围，比如数据层、控制层、视图层等等，视项目不同而不同。

#### 12.2.4 主题（Subject）
[强制] subject是commit目的的简短描述，不超过50个字符：
- 优先使用英文，也可以使用中文
- 结尾不需要标点符号

#### 12.2.5 提交方式
- 所有项目的 MR 都强制要求 MR 方式为 rebase & squash，所以每个 MR 提交到默认分支之后，只会有 1 个 commit
- 大家务必一个 MR 只做一件事，一件事只提一个 MR

#### 12.2.6 分支命名规范
- 把分支名定义为 `feature/{模块}/add-user` 的样式
- 把 MR 和 commit 的 title 定义为 `[feature](模块): 功能描述` 的格式

#### 12.2.7 提交示例
```
[feature](family): 添加族谱可视化功能
[bugfix](auth): 修复用户登录验证问题
[performance](api): 优化族谱查询性能
[refactor](models): 重构用户模型结构
[format]: 代码格式化调整
[docs]: 更新API文档
[test]: 添加用户认证单元测试
[chore]: 更新依赖包版本
```

#### 12.2.8 代码规范要求
- 单元测试覆盖率 ≥ 70%

#### 12.2.9 代码审查
- 所有 MR Approver，请务必严格按照上述要求作 Code Review
